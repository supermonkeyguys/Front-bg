
## 语义化( Semantic Markup ) 


语义化是指使用具有明确含义的 `HTML` 标签来描述内容的结构和作用，而不是仅仅为了布局和样式

**例子：**

语义化标签能清晰表达 "这部分是导航"、"这是文章主体"、"这是页脚" 等信息

```html
<!-- 非语义化（过去常见） -->
<div id="header">...</div>
<div class="nav">...</div>
<div class="content">...</div>
<div class="footer">...</div>

<!-- 语义化（HTML5 推荐） -->
<header>...</header>
<nav>...</nav>
<main>
  <article>...</article>
  <aside>...</aside>
</main>
<footer>...</footer>
```


### 为什么会出现 HTML5 ？解决了什么问题

**HTML5 的出现主要是为了解决 Web 技术在 2000 年代中后期面临的瓶颈和混乱**

具体包括：

1. 缺乏统一标准：
	- 早期 Web 依赖 Flash 、Silverlight 等插件实现音视频和动画，兼容性差、性能低、安全风险高
	- 浏览器厂商各自为政，HTML4 和 XHTML 规范滞后
2. 结构混乱，可访问性差：
	- 开发者大量使用 `<div>` 和 `<span>` 构建页面，导致页面难以理，对搜索引擎 (SEO) 和 屏幕阅读器（无障碍访问）不友好
3. Web 应用能力不足：
	- 传统 HTML 无法满足复杂的 Web 应用（如在线文档、实时聊天、游戏）的需求，缺少本地存储、离线运行、设备访问等能力
4. 移动互联网的兴起：
	- 需要一种能在手机、平板、pc 上统一运行的 Web 技术，HTML5 提供了响应式、跨平台的基础


#### 总结

- **HTML5 是现代 Web 的基石**，让网页从“静态文档”进化为“功能丰富的应用”。
- **语义化** 提升了代码可读性、SEO 效果和无障碍访问能力。
- 它的出现是为了解决 **插件依赖、结构混乱、功能缺失、跨平台兼容** 等核心问题，推动 Web 向更开放、更强大、更标准化的方向发展。

### 添加了哪些语义化标签

`<header>` `<nav>` `<main>` `<article>` `<section>` `<aside>` `<footer>` `<mark>` `<time>` 

#### 语义化标签好处

- SEO 友好：搜索引擎能通过语义化标签快速识别页面结构，提升关键词排名
- 无障碍友好：屏幕阅读器可以通过语义标签向视障用户清晰描述内容


### SEO

#### SEO 优化方法

**SEO (Search Engine Optimization，搜索引擎优化)** 

是指通过 优化网站结构、内容、、技术及外部链接 等手段，提升网站在搜索引擎（如：百度、Google）自然搜索结果中的排名，从而获取更多免费、精确的数字营销方式。其核心目标就是让用户在搜索特定关键词时，目标网站能更靠前地展示，最终提升用户访问量和业务转化

- **网站结构与 URL**：采用扁平化结构（首页 -> 分类 -> 详细页不超过3层），URL 用关键词的拼音 / 英文，避免动态参数堆砌

- **索引控制**：用 `robots.txt` 屏蔽后台、购物车等无效页面，通过 XML sitemap 提交核心页面，加速收录；对重复内容（如：PC / 移动端相同页）用 `canonical` 标签指定主 URL，避免权重分散

- **页面技术优化**：
	- 必做 TDK： Title 含核心词（30字以内），Description 写吸引点击的摘要（120 字内）
	- 语义化标签： 用 `h1`（唯一，放心页面核心标签）、`article`、`section` 等帮助爬虫理解结构
	- 速度优化：图片加 alt 属性 + 压缩，JS / CSS 打包压缩，首屏加载速度控制在 3 秒以内（用 CDN 和 懒加载）


#### SPA 缺点

**SPA 是前端主流应用架构之一，核心特点是 整个应用只有一个 HTML 页面，后续交互仅通过 JS 动态更新页面内容，无需重新加载整个页面**

- 首屏加载速度慢：首次打开需加载完整的 JS / CSS 包（若包体积大，会出现 "白屏"）
- SEO 不友好：初始 HTML 为空，内容靠 JS 动态生成，搜索引擎爬虫（如 百度 / Google）早期难以抓取内容（需要额外做 SEO 优化，如渲染、SSR）
- 前进 / 后退 功能需手动处理：浏览器原生前进后退会触发 URL 变化，但 SPA 需要自己监听 `popstate` 事件、匹配对应内容，否则会出现 "URL 变了但内容没更新" 的问题

### CSR 

CSR 全称是 Client-Side Rendering（客户端渲染），是现代前端开发中主流的页面渲染方式，核心特点是 页面的 HTML 内容在用户浏览器（客户端）中动态生成，区别在于 SSR (服务端渲染) 在服务器端生成完整的 HTML

CSR 的渲染过程主要分为 4步，核心是 "浏览器接管渲染工作"：

1. 用户输入 URL 并发送请求
2. 服务器仅返回一个 "空壳 HTML"（包含基础结构和引入 JS / CSS 的标签，无实际内容）
3. 浏览器下载并执行 JS 代码（如 Vue / React 的打包文件），同时通过 AJAX / Fetch 向服务器请求页面内容所需数据（如 列表数据、用户信息）
4. JS 代码在浏览器中 "拼接" 数据和页面数据，动态生成 DOM 并渲染到页面，最终呈现完整页面


### SSR

SSR 全称是 Server-Side Rendering（服务端渲染），是一种页面渲染方式，核心特点是 页面内容是在服务端生成，再完整发送给客户端，区别在于 CSR（客户端渲染）在浏览器中动态生成内容.

SSR 的渲染过程可分为 5 步，核心是 "服务器提前完成渲染"

1. 用户输入 URL 并发送请求
2. 服务器接收请求后，先调用 API 获取页面所需的数据（如商品详细、文章内容）
3. 服务器将数据传入前端组件（如 React 组件），通过 `ReactDOMServer.renderToString()` 等方法，在服务器端生成完整的 HTML 字符串（包含真实内容，而非空壳）
4. 服务器将生成的 HTML 响应给浏览器，浏览器接受后可直接渲染出页面（首屏快速出现）
5. 浏览器同时下载前端 JS 代码，执行后完成 "水合（Hydration）" ---- 绑定事件、激活交互，让静态页面可变成可交互的动态页面


### SSR 与 CSR 的核心区别



| 维度      | SSR（服务端渲染）           | CSR（客户端渲染）             |
| ------- | -------------------- | ---------------------- |
| 渲染位置    | 服务器                  | 浏览器                    |
| 首屏加载    | 快（直接返回完整 HTML）       | 慢（需下载 JS 并执行后渲染）       |
| SEO 友好度 | 高（HTML 含完整内容，爬虫易抓取）  | 低（初始 HTML 为空，依赖 JS 渲染） |
| 服务器压力   | 大（需动态生成 HTML 和 处理数据） | 小（仅返回静态资源和数据）          |
